# Phase 11 - Portfolio And Job Readiness

This phase turns private practice into public proof.

Before we go further, here are the plain-English words we will use:

- a `portfolio` is a set of proof that shows what you can build and how you think
- an `artifact` is something a reviewer can inspect, such as code, a README, a diagram, a demo, or a write-up
- a `demo` is a short walkthrough that shows the system working
- a `case study` is a written story about a project, the decisions behind it, and what you learned
- a `metric` is a number or score used to measure quality
- `evidence` is proof a reviewer can inspect instead of trusting a claim
- a `hiring manager` is the person who decides whether a team should hire you
- a `recruiter` is the person who helps connect candidates and teams
- an `interview loop` is the sequence of conversations and checks a company uses before hiring

In this phase, the goal is not to decorate GitHub.
The goal is to make another person believe, quickly and fairly, that you can build useful systems, explain them clearly, and support your claims with evidence.

## What This Phase Is Really About

A strong portfolio is an argument.

The argument is simple:

1. I can build real systems.
2. I can measure whether they work.
3. I can explain the tradeoffs honestly.
4. I can talk about the work in a way another person can follow.

## Outcomes

By the end of this phase, you should have:

- three portfolio-grade projects with clear READMEs, evaluations, and demos
- one short narrative that connects the projects into a coherent story
- resume bullets grounded in shipped work
- interview answers for architecture, tradeoffs, failure analysis, and metrics
- a weekly job-search routine that you can repeat

## What This Phase Teaches

Portfolio quality is a technical skill. You are learning to:

- show evidence instead of claims
- explain architecture in a simple sequence
- present metrics honestly
- talk about failures without hiding them
- turn project work into resume, GitHub, and interview language

Think of this phase as the bridge between private practice and public trust.

## Portfolio Standard

Every major project should make these five things obvious:

1. What problem you solved
2. How the system works
3. How you measured it
4. What failed and what changed
5. Why the design choices were reasonable

If a stranger cannot answer those five questions from your repo, the project is not finished.

## The Three Required Proof Types

- build proof: code, folder structure, setup, and architecture
- quality proof: metrics, evals, tests, and error analysis
- communication proof: README, diagram, demo, and short summary

## Time Budget

- Week 1: choose the strongest projects and write the story
- Week 2: improve READMEs, diagrams, and screenshots
- Week 3: write eval summaries and failure stories
- Week 4: convert project evidence into resume and outreach assets

## Final Deliverable

Create a portfolio packet for each project that includes:

- one polished README
- one architecture explanation
- one evaluation summary
- one short demo plan or recording
- one set of interview bullets
